## Context

The `@justeattakeaway/eslint-plugin-snacks-pie-migration` package ships a `snacks-components-data.json` file that maps Snacks component names to their PIE replacements. This file is generated by `scripts/extract-component-data.js`, which scans `packages/components/**/package.json` for entries with `pieMetadata.replaces.snacks`. Today, keeping it in sync is manual.

The repo already has a near-identical automation pattern for icon updates (`update-icons.yml` + `create-icons-update-pr.js` in `pie-monorepo-utils`). This change follows that pattern.

**Existing relevant utilities:**
- `pie-monorepo-utils/utils/find-monorepo-root.js` — walks up from `process.cwd()` to find the repo root via `yarn.lock`
- `pie-monorepo-utils/create-icons-update-pr/create-icons-update-pr.js` — reference implementation for automated PR creation
- `changeset-release.yml` — reference for `slackapi/slack-github-action` usage

## Goals / Non-Goals

**Goals:**
- Automatically detect when `pieMetadata` changes in component packages cause a diff in the generated `snacks-components-data.json`
- Create a branch, changeset, and PR automatically when a diff is found
- Close and clean up stale automation PRs before creating a new one
- Notify `#team-pex-reviews` via Slack when a PR is created
- Provide a `workflow_dispatch` trigger for manual testing

**Non-Goals:**
- Auto-merging the created PR (human review required)
- Detecting changes to other parts of component `package.json` files
- Updating the ESLint rules themselves (only the data file is updated)
- Running on PRs — only runs post-merge on `main`

## Decisions

### D1: Detection via in-memory comparison against the committed JSON

**Decision:** Run `extractComponentData()` in memory and compare its output against the committed `snacks-components-data.json`. If they differ, proceed with automation.

**Rationale:** This avoids any dependency on git diff tooling or file-watching. The committed JSON is the authoritative "last known state"; any drift from it means a component's `pieMetadata` changed in a way that affects the plugin.

**Alternative considered:** Use `git diff` on the component `package.json` files to detect `pieMetadata` changes. Rejected because it requires parsing git diffs and is brittle to merge commit formats.

---

### D2: `extractComponentData` moved to `lib/`, exported as a pure function

**Decision:** Refactor `scripts/extract-component-data.js` into two files:
- `lib/extract-component-data.js` — exports `extractComponentData(startPath)`, no file I/O, no side effects
- `scripts/extract-component-data.js` — thin entry point: calls `lib/`, writes `snacks-components-data.json`

**Rationale:** The detection script in `pie-monorepo-utils` needs to call `extractComponentData` without triggering the file write. Separating pure computation from I/O makes the function reusable and testable.

---

### D3: `startPath` resolved by the caller via `findMonorepoRoot`, not inside the function

**Decision:** `extractComponentData` accepts `startPath` as a parameter. `detect-changes.js` resolves it using `findMonorepoRoot()`:
```js
const root = findMonorepoRoot();
const startPath = path.join(root, 'packages/components');
extractComponentData(startPath);
```

**Rationale:** If `extractComponentData` called `findMonorepoRoot` internally, it would need `pie-monorepo-utils` as a dependency, creating a circular dep since `pie-monorepo-utils` depends on `eslint-plugin-snacks-pie-migration`. Keeping resolution in the caller avoids this entirely.

**Alternative considered:** Using `__dirname`-based path as default. Viable, but using `findMonorepoRoot` in the caller is more explicit and consistent with how other scripts in `pie-monorepo-utils` resolve paths.

---

### D4: `pie-monorepo-utils` depends on `eslint-plugin-snacks-pie-migration` via `workspace:*`

**Decision:** Add `"@justeattakeaway/eslint-plugin-snacks-pie-migration": "workspace:*"` to `pie-monorepo-utils` dependencies. Expose `extractComponentData` via a subpath export `"./extract-component-data": "./lib/extract-component-data.js"`.

**Rationale:** Clean module boundary. The detection script depends on the eslint plugin's extraction logic as a published contract. Using the package name is more maintainable than a brittle relative path.

**Alternative considered:** Relative `require('../../eslint-plugin-snacks-pie-migration/lib/...')` in `detect-changes.js`. Works in a monorepo checkout but breaks the module boundary and is fragile to directory restructuring.

---

### D5: Close stale automation PRs before creating a new one

**Decision:** In `create-eslint-plugin-update-pr.js`, before creating the new PR: list all open PRs, filter by head branch matching `update-eslint-snacks-plugin-*`, close each match, and delete its remote branch.

**Rationale:** Without cleanup, multiple stale PRs accumulate if component changes land before the previous PR is merged. Reviewers should only ever see one PR from this automation at a time.

**Alternative considered:** Skip cleanup and let PRs accumulate. Simpler, but creates reviewer confusion and merge conflicts between automation PRs.

---

### D6: Workflow trigger — push to main with paths filter

**Decision:**
```yaml
on:
  push:
    branches: [main]
    paths:
      - 'packages/components/**/package.json'
  workflow_dispatch:
```

**Rationale:** The `paths` filter avoids starting the job on every merge to `main`. The detection script handles the fine-grained check (only proceeds if `snacks-components-data.json` would change). `workflow_dispatch` enables manual testing.

---

### D7: PR description encodes a structured diff

**Decision:** The `CHANGES_SUMMARY` env var (set by `detect-changes.js`) contains a JSON-serialised diff object with four categories:

| Category | Condition |
|---|---|
| `added` | Key present in current data, absent in stored |
| `removed` | Key present in stored data, absent in current |
| `statusChanged` | Key present in both, `status` field differs |
| `snacksChanged` | Key present in both, `piePackage` differs (renames) |

`create-eslint-plugin-update-pr.js` formats this into Markdown sections in the PR body.

---

### D8: Changeset uses a timestamp-based filename

**Decision:** The changeset file is written as `.changeset/update-eslint-snacks-plugin-${Date.now()}.md` with a `minor` bump.

**Rationale:** Changeset filenames just need to be unique. A timestamp is deterministic for a given run and won't collide across parallel executions.

**Alternative considered:** Using `@changesets/write` to generate a random slug. Adds a dependency; timestamp is simpler.

## Risks / Trade-offs

**[Risk] Stale branch deletion fails due to branch protection** → Branch protection rules typically only apply to `main`. The automation branches (`update-eslint-snacks-plugin-*`) are unprotected, so deletion should always succeed with the GitHub App token.

**[Risk] Race condition: two merges to main in quick succession both trigger the workflow** → GitHub Actions concurrency is not set to cancel-in-progress for push events in this workflow. The second run will attempt to close the PR created by the first run and create a new one. This is the correct behaviour and is safe because each run closes stale PRs before creating. Add `concurrency` group on `refs/heads/main` to serialize if needed.

**[Risk] `yarn install` in CI takes time on every merge** → The workflow reuses the setup pattern from `update-icons.yml` which already runs `yarn install`. No additional cost beyond what icons automation already pays.

**[Risk] `snacks-components-data.json` subpath export breaks existing consumers** → The `exports` field change from a string to an object is backward-compatible: `"."` still resolves to `./lib/index.js`, so existing `require('@justeattakeaway/eslint-plugin-snacks-pie-migration')` calls are unaffected.
